## 목표

프로젝트의 핵심 구성 요소(페이지, 컴포넌트, API, 라이브러리, 외부 서비스)를 분석하고,
Mermaid `graph TD` 다이어그램과 함께 구조 설명을 제공한다.

## 분석 단계

### 1단계: 프로젝트 성격 파악

아래 파일을 우선 읽어 프레임워크와 주요 의존성을 파악한다:

- `package.json` — 프레임워크, 주요 라이브러리
- `README.md` — 프로젝트 개요 (존재하는 경우)
- 최상위 디렉토리 구조

### 2단계: 핵심 구성 요소 탐색

프레임워크에 맞게 탐색한다.

**Next.js App Router**
- `app/` — 페이지 및 레이아웃 (라우팅 구조)
- `app/api/` — API 엔드포인트
- `components/` — UI 컴포넌트 (하위 폴더별 역할 파악)
- `lib/` 또는 `utils/` — 유틸리티 및 비즈니스 로직

**Next.js Pages Router**
- `pages/` — 페이지
- `pages/api/` — API 라우트

**일반 React / Node**
- `src/` 하위 구조 탐색

### 3단계: 관계 파악

주요 파일의 import 패턴을 확인해 컴포넌트 간 의존 관계를 파악한다.
모든 import를 추적하기보다는 아키텍처 이해에 중요한 흐름에 집중한다.

### 4단계: 다이어그램 작성

**노드 표기 규칙**

| 종류 | Mermaid 표기 | 예시 |
|------|-------------|------|
| 페이지/라우트 | `[페이지명]` | `[홈페이지]` |
| 컴포넌트 | `(컴포넌트명)` | `(Header)` |
| API 엔드포인트 | `[[API명]]` | `[[/api/auth]]` |
| 외부 서비스/DB | `[(서비스명)]` | `[(Database)]` |
| 레이아웃 | `{레이아웃명}` | `{RootLayout}` |

**작성 원칙**
- 노드는 30개 이하로 유지한다. 비슷한 역할은 그룹으로 묶는다.
- 모든 파일을 나열하지 말고 아키텍처 이해에 중요한 것만 포함한다.
- 추측하지 말고 실제 코드를 읽어 사실에 근거해 그린다.

### 아키텍처 개요

프로젝트의 전체 구조를 2~4문장으로 설명한다:
- 사용 중인 프레임워크 및 주요 스택
- 주요 영역(페이지, API, 컴포넌트)의 구성 방식
- 데이터 흐름의 큰 그림
